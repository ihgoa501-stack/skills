# MoSCoW Method

**Origin:** Dai Clegg (1994), Oracle UK — used extensively in Agile and PRINCE2

**Core idea:** Categorize every requirement or task into one of four buckets to create a shared understanding of priority, with clear non-negotiables.

## The Four Categories

| Category | Meaning | What Happens If Missing |
|----------|---------|------------------------|
| **M** — Must have | Non-negotiable, delivery fails without it | Project is considered failed |
| **S** — Should have | Important but a workaround exists | Painful, but acceptable delay |
| **C** — Could have | Desirable, nice-to-have | Lower impact, easy to defer |
| **W** — Won't have | Explicitly out of scope for now | Intentionally deferred |

## Typical Distribution (Agile Projects)

```
  Must ──── ~40%
  Should ── ~30%
  Could ─── ~20%
  Won't ─── ~10%
```

## How to Use

1. **List** all requirements, features, or tasks
2. **Sort each** into M / S / C / W
3. **Reality check:** If everything is "Must," nothing is. Force hard tradeoffs.
4. **Communicate W** explicitly — "Won't have" is a positive decision, not neglect

## Common Pitfalls
- ❌ Everything labeled "Must" — defeats the purpose
- ❌ Confusing "Must" with "Would be nice" — Must means the project is dead without it
- ❌ Not using "Won't have" explicitly — unspoken expectations cause friction later
- ❌ Too many stakeholders with equal veto power — prioritize by value, not by loudness

## When to Use
- Defining MVP scope
- Sprint planning with too many items
- Product roadmap prioritization
- Stakeholder alignment sessions
