# Deliberate Practice (刻意练习)

**Origin:** Anders Ericsson, *Peak: Secrets from the New Science of Expertise* (2016) — the science behind the "10,000-hour rule," but 10,000 hours is a correlation, not a cause.

## Core Principle

> **Not all practice makes perfect.** Repeating what you already know is *naive practice*. Deliberate practice targets your specific weaknesses with focused effort and immediate feedback.

## Deliberate vs Naive Practice

| | Naive Practice | Deliberate Practice |
|---|---------------|-------------------|
| **Goal** | Do the activity | Improve a specific sub-skill |
| **Difficulty** | Comfortable | At the edge of current ability |
| **Feedback** | None or delayed | Immediate, specific |
| **Mental effort** | Low — autopilot | High — fully engaged |
| **Duration** | As long as you want | 1–4 hours max (mental exhaustion) |
| **Result** | Plateau within months | Continuous improvement |

## The 5 Elements of Deliberate Practice

1. **Well-defined, specific goal** — Not "get better at coding" but "reduce bug rate in PR reviews by spotting concurrency issues"
2. **At the edge of comfort** — If you can do it easily, it's not practice. It should feel effortful.
3. **Immediate, specific feedback** — Tests that fail = feedback. Code review feedback = feedback. A recording of your performance = feedback.
4. **Repetition with refinement** — Do it again, incorporating the feedback. Then again.
5. **Mental representation** — Build richer mental models of what "good" looks like. Study masters' output.

## How to Apply to Knowledge Work

| Domain | Deliberate Practice Example |
|--------|---------------------------|
| **Coding** | Solve the same problem 3 different ways → compare performance/readability |
| **System design** | Redesign an existing system from scratch → compare with the original → understand the tradeoffs |
| **Writing** | Write a paragraph, cut it by 50% while keeping all meaning. Repeat. |
| **Debugging** | Time-box yourself: "find the root cause in 30 min." Review what you missed. |
| **Presenting** | Record yourself, watch at 2x speed, fix one thing. Repeat. |

## The "Teacher / Coach" Problem

Deliberate practice works best with a coach who:
- Designs the right drills for your level
- Provides immediate, accurate feedback
- Pushes you past what's comfortable

Without a coach, you must be your own coach:
- **Use tools** — linters, benchmarks, test suites, automated feedback
- **Study masters** — read great code, watch experts work, reverse-engineer excellence
- **Peer review** — the closest thing to a coach in knowledge work

## Anti-Patterns

- ❌ **The 10,000-hour myth** — Hours count for nothing if they're mindless repetition
- ❌ **Grinding without feedback** — Coding alone for years with no code review = plateau, not expertise
- ❌ **Always staying comfortable** — You get what you always got. Real practice feels frustrating.
- ❌ **No rest** — Deliberate practice is mentally exhausting. 4 hours/day is the upper limit for most people.

## When to Use

- You've plateaued in a skill despite doing it regularly
- You want to accelerate expertise beyond "years of experience"
- Designing a learning plan for yourself or your team
- Evaluating training programs — are they deliberate practice or naive practice?
