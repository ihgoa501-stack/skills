# OKR (Objectives and Key Results)

**Origin:** Andy Grove (Intel) → John Doerr → Google

## Structure

| Component | Description | Example |
|-----------|-------------|---------|
| **Objective** | Qualitative, inspirational, time-bound | "Deliver a world-class mobile experience" |
| **Key Results** | 3–5 quantitative, measurable outcomes | "Achieve 4.5+ app store rating" |

## Writing Good Key Results

| ❌ Bad KR | ✅ Good KR |
|-----------|-----------|
| "Improve user engagement" | "Increase DAU/MAU from 0.3 to 0.5" |
| "Fix all critical bugs" | "Reduce P0 bug count from 12 to 0" |
| "Launch the new feature" | "Ship v2.0 to 100% of GA users by EOD Q2" |
| "Make the site faster" | "Reduce LCP from 4.2s to under 2.5s (p75)" |

## Rules
- OKRs should be **ambitious** — 70% completion is success
- Set KRs **quarterly**, high-level Objectives annually
- KRs must have a clear **numeric measure**
- Keep **3–5 KRs** per Objective

## Common Pitfalls
- ❌ KRs that are tasks, not outcomes ("Ship the report feature")
- ❌ Too many KRs
- ❌ KRs without metrics ("Make it better")
- ❌ Setting only safe, 100%-achievable goals

## Cascading
- **Company OKRs** → **Team OKRs** → **Individual OKRs**
- Lower-level OKRs should connect to higher-level ones
