## Personal Productivity

### GTD (Getting Things Done)

**Origin:** David Allen, *Getting Things Done* (2001)

Five core steps:

| Step | Description |
|------|-------------|
| **Collect** | Capture everything that has your attention into an inbox |
| **Process** | If it takes ≤2 min, do it now; otherwise defer, delegate, or delete |
| **Organize** | Sort into next actions, projects, waiting-for, calendar, someday/maybe |
| **Review** | Weekly review to keep the system current and trustworthy |
| **Engage** | Choose what to work on based on context, time, energy, priority |

**Key insight:** Clear your mind by externalizing all tasks — your brain is for having ideas, not holding them.

### Pomodoro Technique (番茄工作法)

**Origin:** Francesco Cirillo, late 1980s

| Phase | Duration |
|-------|----------|
| Focus session | **25 min** uninterrupted work |
| Short break | **5 min** rest |
| Every 4th cycle | **15–30 min** long break |

Rules:
- A Pomodoro is indivisible — once started, must finish
- Interruptions reset the current Pomodoro
- Tasks >5–7 Pomodoros should be split
- Tasks <1 Pomodoro should be batched

**Best for:** Deep focus tasks, procrastination reduction, time boxing.

### Time Management Quadrant (四象限法则 / Eisenhower Matrix)

**Origin:** Stephen Covey, *The 7 Habits of Highly Effective People*

| | **Urgent** | **Not Urgent** |
|---|-----------|----------------|
| **Important** | I. **Do now** (crises, deadlines) | II. **Schedule** (planning, relationships, exercise) |
| **Not Important** | III. **Delegate / batch** (interruptions, busywork) | IV. **Eliminate** (time wasters, trivia) |

**Key insight:** Spend most energy on **Quadrant II** (important but not urgent) — neglecting it turns strategic work into crises.

### Atomic Habits (原子习惯)

**Origin:** James Clear, *Atomic Habits* (2018)

Four laws of behavior change:

1. **Make it obvious** — design environment cues
2. **Make it attractive** — pair with something enjoyable
3. **Make it easy** — reduce friction, 2-minute rule
4. **Make it satisfying** — immediate reward

**Core idea:** 1% improvement every day compounds to ~37× better over a year.

## Problem Solving

### 5WHY Analysis (五问法)

**Origin:** Sakichi Toyoda / Toyota Production System

Ask "Why?" repeatedly (typically 5 times) to drill from symptom to root cause.

**Classic example — machine stopped:**

| Level | Question | Answer |
|-------|----------|--------|
| 1 | Why did the machine stop? | Fuse blew (overload) |
| 2 | Why overload? | Bearing insufficient lubrication |
| 3 | Why insufficient lubrication? | Oil pump failed |
| 4 | Why pump failed? | Shaft worn out |
| 5 | Why shaft worn? | **No filter — debris entered** |

**Fix:** Install an oil filter, not just replace the fuse.

Three analysis dimensions:
1. **Why it happened** — physical root cause (source)
2. **Why it wasn't detected** — inspection gap
3. **Why not prevented** — system/process gap

**Rules:** Base on facts, not blame. Stop when a fixable root cause is found (not always exactly 5).

### PDCA Cycle (Plan-Do-Check-Act / 戴明环)

**Origin:** Walter Shewhart / W. Edwards Deming

| Phase | Action |
|-------|--------|
| **Plan** | Define the problem, analyze current state, set target, design solution |
| **Do** | Implement the solution on a small scale / pilot |
| **Check** | Measure results against targets, analyze gaps |
| **Act** | Standardize if successful; otherwise adjust and repeat |

**Best for:** Continuous improvement, process optimization, quality management.

## Goal Setting

### OKR (Objectives and Key Results)

**Origin:** Intel (Andy Grove) → Google (John Doerr)

| Component | Description | Example |
|-----------|-------------|---------|
| **Objective** | Qualitative, inspirational, time-bound goal | "Deliver a world-class mobile experience" |
| **Key Results** | 3–5 quantitative, measurable outcomes | "Achieve 4.5+ app store rating" |

Rules:
- OKRs are **ambitious** (70% completion is success)
- Set **quarterly** for KRs, annually for high-level Objectives
- KRs must have a clear **numeric measure**

### SMART Principle

**Origin:** George T. Doran (1981)

| Letter | Meaning | Question |
|--------|---------|----------|
| **S** | Specific | What exactly? Who? Where? |
| **M** | Measurable | How will I know it's done? |
| **A** | Achievable | Is it realistic with available resources? |
| **R** | Relevant | Does it align with broader goals? |
| **T** | Time-bound | When must it be completed? |

**Best for:** Setting clear, actionable goals at any level.

## Team & Development

### Agile Manifesto (敏捷开发)

Four values:
1. **Individuals and interactions** over processes and tools
2. **Working software** over comprehensive documentation
3. **Customer collaboration** over contract negotiation
4. **Responding to change** over following a plan

### Scrum

| Ceremony | Frequency | Duration | Purpose |
|----------|-----------|----------|---------|
| Sprint Planning | Start of sprint | 2–4 hrs | Define sprint goal & backlog |
| Daily Standup | Daily | 15 min | Sync, blockers, plan |
| Sprint Review | End of sprint | 1–2 hrs | Demo completed work |
| Retrospective | End of sprint | 1–1.5 hrs | Inspect & adapt process |

**Roles:** Product Owner, Scrum Master, Development Team.
**Artifacts:** Product Backlog, Sprint Backlog, Increment.

### Kanban (看板)

Principles:
- **Visualize** workflow (columns: To Do → In Progress → Done)
- **Limit WIP** (Work In Progress) — finish before starting new
- **Manage flow** — measure cycle time, optimize throughput
- **Make policies explicit** — definition of done, pull criteria

**Best for:** Continuous delivery, operations teams, support workflows.

### XP (Extreme Programming / 极限编程)

Core practices:
- Test-Driven Development (TDD)
- Pair Programming
- Continuous Integration
- Collective Code Ownership
- Small, frequent releases
