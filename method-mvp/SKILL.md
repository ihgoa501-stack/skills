# MVP — Minimum Viable Product (最小可行产品)

**Origin:** Eric Ries, *The Lean Startup* (2011)

## Core Definition

> An MVP is the **smallest thing** you can build that lets you **learn** whether a core assumption is true — with the **least effort**, not the fewest features.

An MVP is **not** a half-baked product. It's the fastest experiment that delivers a valid result.

## What MVP Is (and Isn't)

| ✅ Is | ❌ Isn't |
|------|----------|
| The fastest path to validated learning | Version 0.1 of the full vision |
| A deliberate experiment with a hypothesis | An excuse to ship low quality |
| Designed to answer one key question | Designed to please everyone |
| Discarded or pivoted based on results | A commitment to the initial direction |

## Build-Measure-Learn Loop

```
Build → Measure → Learn → (Pivot or Persevere) → Build → ...
```

| Phase | Key Question |
|-------|-------------|
| **Build** | What's the smallest thing we can make to test our riskiest assumption? |
| **Measure** | What data proves or disproves our hypothesis? |
| **Learn** | Do we persevere (evidence supports) or pivot (evidence refutes)? |

## MVP Types — Choose the Right One

| Type | How It Works | Best For |
|------|-------------|----------|
| **Landing Page MVP** | A page describing the product + signup — gauge interest before building | New product ideas, demand validation |
| **Concierge MVP** | Manually deliver the service to one customer, then automate | Service-heavy products, complex workflows |
| **Wizard of Oz MVP** | Users think it's automated, but you're doing it manually behind the scenes | AI/automation products, testing UX before engineering |
| **Single Feature MVP** | Ship only the one feature that differentiates your product | Established markets, clear problem |
| **Video MVP** | A demo video showing the vision — measure reactions | Hardware, hard-to-build products |
| **Piecemeal MVP** | Combine existing tools (Airtable + Zapier + Stripe) instead of building | B2B, workflow automation |

## How to Define Your MVP

1. **State your riskiest assumption** — What must be true for this to work? (e.g., "People will pay for X")
2. **Define the success metric** — What number proves it? (e.g., "20% signup-to-paid conversion")
3. **Identify the minimum to test it** — What's the absolute simplest way to get that data?
4. **Cut everything else** — Features that don't directly test the assumption are waste
5. **Set a deadline** — Time-box the experiment to avoid endless tinkering

## Anti-Patterns

- ❌ **MVP = MBP (Maximum Bloated Product)** — "Just one more feature before we launch"
- ❌ **No hypothesis** — Building without knowing what you're trying to learn
- ❌ **Ignoring the data** — When results contradict the vision but you ship anyway
- ❌ **Fake MVP** — Building the full product but calling the first release "MVP" to excuse gaps
- ❌ **Survivor-bias MVP** — "Dropbox/DoorDash did it with a video, so we should too" — pick the *right* type for your context

## When to Use

- Building something new in an uncertain market
- Stakeholders want "everything in V1" — need a framework to push back
- You're spending months building but haven't shown it to a single user
- Investors/leadership ask "what's your MVP?" and you need a real answer
- Before writing any code for a new product idea

## When NOT to Use

- Incremental improvements to an existing, validated product
- Regulatory/safety-critical systems where "minimum" is inappropriate
- Pure research projects where learning is the product
- You already have paying customers with clear feature requests
